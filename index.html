1 · Architecture — structure, runtime, application logic

Three lenses on one system. The structure is the static split — how the srsRAN gNB decomposes along O-RAN boundaries (a Central Unit, CU-CP + CU-UP; a Distributed Unit, RLC / MAC / scheduler + PHY; a Radio Unit), joined by the F1 / E1 / FAPI / OFH interfaces. The runtime is the execution substrate — how code runs (executors, worker threads, the slot-driven loop in the gNB; the event loop in the Open5GS core). It is application-logic-independent and the more fundamental layer: it schedules and dispatches without knowing what it runs. The application logic is what the runtime executes — the domain behaviour (NR scheduling, the core's state machines), modelled as actor graphs and mapped to our primitives. Keeping the runtime and the application logic apart is the point: one substrate, many tenants.

F1 — CU ↔ DU

The 3GPP CU/DU split (TS 38.470 family). F1-C carries control (F1AP) between CU-CP and DU; F1-U carries user data (GTP-U) between CU-UP and DU. This is the primary disaggregation boundary and the one srsRAN already supports as a split deployment.

E1 — CU-CP ↔ CU-UP

The control/user-plane separation inside the CU (TS 38.460 family, E1AP). Lets the control plane and a scaled-out user plane evolve independently — a natural place to study placement and scaling.

FAPI — MAC ↔ PHY

The Small Cell Forum interface (SCF 222) between the scheduler in DU-high and the PHY in DU-low (P5 config, P7 data). The seam where AI-RAN inference and L1 acceleration attach, under hard real-time deadlines.

Open Fronthaul 7.2x — DU ↔ RU

The O-RAN WG4 lower-layer split (7.2x) between DU-low and the RU, over eCPRI / RoE. srsRAN provides an O-RU-facing fronthaul path; the timing budget here dominates deployability.

Runtime — the execution substrate

The runtime is the application-logic-independent execution substrate — how code runs, not what it computes — and the more fundamental layer. Read from the code, both stacks reduce to one dispatch loop (below): a source feeds work items — the scheduling unit — through a queue to a worker, which dispatches each to a handler (the boundary into application logic); a timer supplies deferred work. srsRAN and Open5GS each instantiate this loop — the comparison shows it is the same loop in two regimes.

srsRAN runtime

The DU-high runtime is a set of executors the du_high_executor_mapper hands out — one per concern. A layer never touches a thread or a queue: it is given an executor, calls execute(task), and the mapper has already decided where that task runs.

Components

The runtime is seven primitives that compose in one direction. A producer holds an executor and calls execute(task). The task is a unique_task — a type-erased void() closure, the only currency the runtime moves. If that executor is a strand, it serialises the producer's tasks against each other — lock-free, one runner at a time — before forwarding. The executor drops the task into a priority queue; a worker thread pops it and runs it. A timer re-posts expired work onto an executor, so deferred tasks re-enter the same path. The mapper is the directory: it owns the wiring and hands each producer its executor. The diagram traces this path; the table names the parts. Nothing in it knows what the task computes — which is exactly why the layers are tenants, not components.

Each one, in the order work flows through them:

Task — unique_task
constexpr size_t default_unique_task_buffer_size = 64;

// generic moveable task
using unique_task = unique_function<void(), default_unique_task_buffer_size>;

The unit of work, and the only thing the runtime moves: a type-erased void() closure with a 64-byte inline buffer, so a slot lambda that captures a few pointers never allocates. An executor accepts it; a worker runs it and clears it, releasing the captures. (adt/unique_function.h)

Executor — task_executor
class task_executor {
public:
  virtual ~task_executor() = default;
  [[nodiscard]] virtual bool execute(unique_task task) = 0;   // run inline if already on-context
  [[nodiscard]] virtual bool defer(unique_task task)   = 0;   // always enqueue for later
};

The pure interface — two virtuals, nothing else. Every concrete part below (the strand, the worker pool) is a task_executor; a tenant holds one and calls it. The [[nodiscard]] bool is the back-pressure the hot path reads to skip a slot rather than block. (support/executors/task_executor.h)

Strand — task_strand
bool on_new_task(enqueue_priority) {
  return job_count.fetch_add(1U, std::memory_order_acq_rel) == 0U;   // 0 → 1: I am the runner
}
...
bool handle_enqueued_task(enqueue_priority prio) {
  bool strand_acquired = state.on_new_task(prio);
  if (not strand_acquired) { return true; }         // another thread already holds the strand
  ... get_task_executor_ref(out_exec).defer([this]() { run_enqueued_tasks(); });   // winner drains batch
}

Serialises one producer's tasks without a mutex. The thread that moves the atomic counter 0 → 1 wins and defers run_enqueued_tasks to the wrapped executor; everyone else only enqueues and returns. One runner at a time — this is how a cell's slots stay ordered while cells run in parallel. A strand is itself a task_executor, so it stacks transparently in front of the worker pool. (support/executors/strand_executor.h)

Priority queue — priority_task_queue
class priority_task_queue {
public:
  [[nodiscard]] bool try_push(task_priority prio, unique_task task);
  [[nodiscard]] bool try_pop(unique_task& t);     // scans high → low priority
private:
  std::vector<std::unique_ptr<detail::any_task_queue>> queues;   // one sub-queue per priority
};

A vector of per-priority sub-queues. try_push routes a task to its bucket; try_pop scans highest to lowest, so a slot indication always overtakes background work. Executors push into it; the worker pops from it. (support/executors/detail/priority_task_queue.h)

Worker — priority_task_worker
void priority_task_worker::run_pop_task_loop() {
  auto consumer = task_queue.create_consumer();
  unique_task t;
  while (consumer.pop_blocking(t)) {   // block until a task arrives
    t();                               // run it
    t = {};                            // release the task's captures
  }
}

A dedicated thread, and the place a task finally executes: block on the queue, pop in priority order, run the void(), clear it. Because it only ever sees a unique_task, it cannot name the MAC — the scheduler is just a tenant whose work happens to be next. (lib/support/executors/priority_task_worker.cpp)

Timer — timer_manager / unique_timer
unique_timer timer_manager::create_unique_timer(task_executor& exec);   // bind a timer to an executor
...
// on expiry the manager posts the callback as a task onto that executor:
return timer.frontend->exec->defer([frontend, expiry_epoch]() {
  if (frontend->backend_ch.current_cmd_id() == expiry_epoch)
    frontend->timeout_callback(frontend->id);
});

The one component that creates work rather than carrying it. tick() advances a time wheel; when a timer expires, the manager defers its callback onto the executor the timer was bound to — so a timeout re-enters the runtime as an ordinary task, on the same queues and workers. (support/timers.h, lib/support/timers.cpp)

Mapper — du_high_executor_mapper
class du_high_executor_mapper {
public:
  virtual du_high_cell_executor_mapper& cell_mapper() = 0;   // executors, per cell
  virtual du_high_ue_executor_mapper&   ue_mapper()   = 0;   // executors, per UE
  virtual task_executor& du_control_executor() = 0;
  virtual task_executor& f1c_rx_executor()     = 0;
  ...
};

The directory. It owns the strand-and-worker topology and answers one question — “which executor for this producer?” — by cell, by UE, or by control / I-O concern. Every handle it returns is a task_executor, so a tenant receives a seam, never a thread. Re-shaping the topology (dedicated workers vs. a shared strand pool) is a change here, not in any tenant. (du/du_high/du_high_executor_mapper.h)

The layers are not components. MAC, RLC, the F1 interfaces, the DU control plane — they sit outside the runtime as tenants. Each is handed one of these executors by the mapper and only ever calls execute(); the runtime never learns what it computes. Which executor each layer is given — at which priority, on which strand — is the boundary between application and runtime, and it belongs with the application logic, not here.

The mechanism

The hot path is a deadline. The clock hands each cell its slot, and the MAC enqueues that slot's work on the cell's high-priority executor — and drops it if the queue is full. That branch is the hard real-time deadline, in code (lib/mac/mac_dl/mac_cell_processor.cpp):

void mac_cell_processor::handle_slot_indication(const mac_cell_timing_context& context) {
  if (not slot_exec.execute([this, context] {
        handle_slot_indication_impl(context.sl_tx, ...);     // → scheduler → sched_result
      })) {
    logger.warning("Skipped slot indication={}. Cause: DL task queue is full.", context.sl_tx);
  }
}

Inside, the work reaches the scheduler synchronously and returns the result by reference — no copy, no allocation on the path (scheduler_impl.cpp):

const sched_result& scheduler_impl::slot_indication(slot_point sl_tx, du_cell_index_t cell) {
  cell.run_slot(sl_tx);
  return cell.last_result();
}

From there the path is just the components above, in sequence. The slot lambda is a Task; the MAC hands it to the cell's high-priority Executor; that cell's Strand keeps its slots ordered while letting other cells run; a Worker finally pops and runs it. None of them knows it is a MAC slot — the executor sees a void(), so the scheduler is a tenant the runtime could swap without recompiling. Which topology carries the path is a config field, not a rewrite: du_high_executor_config is a variant of a per-cell dedicated_cell_worker list or a strand_based_worker_pool.

What it commits to. One slot is the indivisible quantum, bounded by a hard deadline the code would rather skip than miss; work is type-erased so the substrate is blind to its tenants; ordering is bought with a lock-free strand, not a lock. These three are what a unified runtime must preserve for the RAN.

Open5GS runtime

One network function's single-threaded event loop (Open5GS v2.7.7; AMF shown). The loop body is amf_main (src/amf/init.c): it blocks in ogs_pollset_poll with a timeout set to the next timer's expiry (ogs_timer_mgr_next), fires any due timers (ogs_timer_mgr_expire), then drains the queue — ogs_queue_trypop — and hands each ogs_event to ogs_fsm_dispatch, which calls the current state's handler (e.g. amf_state_operational) and, on an OGS_FSM_TRAN, runs the exit / entry transitions. A work source — an SCTP/SBI socket going readable (kqueue here), or a timer — becomes an event via ogs_queue_push, waking the pollset through a notify fd. Scheduling unit = one ogs_event. The FSM engine is substrate; the states it runs are the tenant.

The runtime is fifteen lines. A network function's entire execution model is its *_main reactor — for the AMF, src/amf/init.c:

static void amf_main(void *data) {
  ogs_fsm_t amf_sm;
  ogs_fsm_init(&amf_sm, amf_state_initial, amf_state_final, 0);
  for (;;) {
    ogs_pollset_poll(ogs_app()->pollset,
                     ogs_timer_mgr_next(ogs_app()->timer_mgr));   // block until I/O OR the next timer
    ogs_timer_mgr_expire(ogs_app()->timer_mgr);                  // fire due timers → may enqueue
    for (;;) {
      amf_event_t *e = NULL;
      int rv = ogs_queue_trypop(ogs_app()->queue, (void**)&e);
      if (rv == OGS_DONE)  goto done;                            // shutdown
      if (rv == OGS_RETRY) break;                                // queue empty → re-poll
      ogs_fsm_dispatch(&amf_sm, e);                             // run to completion
      ogs_event_free(e);
    }
  }
done: ogs_fsm_fini(&amf_sm, 0);
}

The poll timeout is the next timer's expiry, so one thread keeps millisecond timer fidelity with no timer thread. The in-source comment even pins an invariant: expire timers before draining the queue, so a timer and a message racing for the same instant resolve by wall-clock.

State is a function pointer. An ogs_fsm_t is three handlers, a transition is one assignment, and dispatch runs the current state then detects the change to fire exit/entry (lib/core/ogs-fsm.c):

void ogs_fsm_dispatch(void *fsm, void *event) {
  ogs_fsm_t *sm = fsm;  ogs_fsm_handler_t tmp = sm->state;
  (*tmp)(sm, event);                          // run current state with the event
  if (sm->state != tmp)                        // a handler called OGS_FSM_TRAN?
    fsm_change(fsm, tmp, sm->state, event);    // exit old state, enter new
}

Every context — the AMF, each gNB, each UE — carries its own ogs_fsm_t, and a state handler re-dispatches the event into the child machine. Control flow is a tree of (state × event) → handler, not a call stack.

One thread, run to completion, no locks. The whole NF is ogs_thread_create(amf_main, NULL). Every input — an NGAP message, an SBI response, a timer — becomes a 256-byte ogs_event_t, is drained from the single queue, and dispatched to completion before the next. Nothing in the NF preempts anything, so nothing takes a lock; the only cross-NF channel is the network (SBI). Race-freedom is bought by giving up intra-NF parallelism.

What it commits to. A single run-to-completion thread (no locks, no intra-NF concurrency); a uniform event as the only unit of work; the FSM as the one control structure. These are what a unified runtime must preserve for the core.

Comparison — two regimes

The two runtimes are that one loop with different data structures in each slot. Each row is a primitive; the columns are how srsRAN and Open5GS realise it — every symbol is a real type read from the source.

Same loop, two regimes. srsRAN is time-driven — a periodic slot on a hard real-time deadline, dispatched at enqueue across a worker pool (a strand per cell), state held by the tenant. Open5GS is event-driven — an aperiodic event, best-effort, dispatched at dequeue on a single thread per NF, state in a generic FSM engine. This runtime is the hub of the research program (see Roadmap).

Toward a unified runtime

Both stacks are the same shape (the model above): submit a work item to a queue; a loop pulls each and routes it to a handler. A unified runtime would expose exactly that — one submission, one loop — and the seam already exists in each: srsRAN's task_executor::execute(unique_task) and Open5GS's ogs_queue_push + ogs_fsm_dispatch are the two halves of it.

runtime.submit(work_item, handler, /* priority or deadline */);   // srsRAN: task_executor::execute
                                                                 // Open5GS: ogs_queue_push
runtime.run();                                                   // pull → dispatch → repeat

What such a runtime must parameterize is everything the two regimes disagree on:

  • Trigger — a periodic clock vs. I/O + timer readiness.
  • Concurrency — a worker pool with lock-free per-cell strands vs. one run-to-completion thread. The hard one.
  • Deadline — a hard per-slot bound that drops late work vs. best-effort that never drops.
  • Dispatch point — routed by cell at enqueue vs. by state at dequeue.
  • State — held by the tenant (the runtime is stateless) vs. an FSM engine the runtime provides.

The conflict that resists unification is concurrency × deadline. srsRAN's correctness requires non-preemptible slot work to finish before the next tick; Open5GS's correctness requires that nothing in an NF runs concurrently, so no handler ever locks. One runtime cannot promise both contracts to the same handler. It can only offer them as two scheduling policies behind one submit/run surface — a real-time strand executor for the RAN, a single-thread reactor for the core — sharing the work item, queue, timer, and dispatch machinery but not the scheduling guarantee. Common mechanism, pluggable scheduling policy is where the redesign cuts.

Application logic — what the runtime executes

The application logic is the domain behaviour — NR scheduling, the core's state machines — that the runtime runs. We model it as actor graphs (events, states, actions), expressed in the runtime's primitives but carrying the domain semantics, and map it one component at a time. It is deliberately kept apart from the runtime — one substrate, many tenants.

MAC scheduler

The first mapped actor graph: the DU-high MAC scheduler (srsRAN_Project release_25_10). The slot tick (slot_indication) is the runtime boundary — the runtime delivers it as the scheduling unit, and everything below is application logic: a per-slot dispatch tree (common scheduling → UE scheduling → policy, time_rr | time_qos, a control knob → channel allocation) that reads state (UE + HARQ, buffers, resource grid) and emits grants (sched_result). Every node maps to a real symbol.

2 · Design

Thesis. The efficiency of the RAN is governed by where the functional boundaries are drawn and how work is scheduled across them. By treating srsRAN's O-CU / O-DU split as the substrate and the F1 / E1 / FAPI / OFH interfaces as movable seams, we can study restructuring and AI-RAN integration as controlled, measurable changes rather than redesigns.

Methodology — how the bring-up was cracked

A 5G stack fails silently in a dozen places. What turned ~15 dead ends into a working link was a discipline, not luck:

  • Isolate layer by layer. Build → core health → gNB-to-AMF → RAN attach → NAS/PDU → data path. Confirm each layer's post-condition before climbing to the next.
  • Configure from the authoritative docs, not by adaptation. Pull the complete official config and reason about the protocol; a stripped-down hand-rolled config silently drops required fields.
  • Reason about the mechanism, not the symptom. "Cell selection failed" → trace MIB → CORESET#0 → SIB1. "HTTP 400 at registration" → SCP NF-discovery → PLMN mismatch.
  • Adversarially verify the fix against the docs before trusting it — this caught a wrong CORESET#0 index that would have re-created the bug.
  • Verify by observable post-condition — a RACH on the air, an assigned IP, 0% ping loss — never by "the config looks right".

1 · DSL design — model the RAN formally

Grounded in the 3GPP spec — two graphs and the contracts that bind them, turned into a running implementation by code generation.

  • Knowledge graph — the static structure: units (CU-CP / CU-UP / DU-high / DU-low / RU), interfaces (E1 / F1 / FAPI / OFH), functions (RRC, PDCP, RLC, MAC, scheduler, PHY) and the dependencies among them. What is connected to what.
  • Actor graph — the dynamic behaviour in three primitives: events (packet arrival, TTI boundary, HARQ feedback), states (queue occupancy, HARQ buffers, channel estimate), and actions (admit a grant, encode a transport block, forward over F1). Each actor is a state machine reacting to events with actions. How the system moves.
  • Event-dispatching tree — how one event propagates to the actors that handle it: the routing hierarchy that turns a TTI boundary or a packet arrival into the ordered actions it triggers.
  • Pre/post-condition determination — the contract on each action: what must hold before it fires (pre-conditions, the guards) and what it guarantees after (post-conditions, the effects) — the hook formal reasoning (5) checks.

2 · Runtime — the actor-based execution environment

The fundamental layer — an actor-based execution environment, the substrate the model runs on and the study's crux. Its central question is the scheduling unit: at what granularity is work decomposed and scheduled — a slot, a TTI, a per-UE function invocation, an actor step? The unit fixes where parallelism, pipelining, and placement across cores and accelerators can live, and is the lever restructuring actually pulls.

3 · Control knobs — the configuration space

Which states and configurations actually move the metrics: numerology / SCS, bandwidth, MCS table, the split point (monolithic / F1 / full O-RAN), scheduler policy, function placement, batching. These are the actor graph's decoupled state — the configuration space the study sweeps, feeding the Performance dimensions directly.

4 · Hardware integration

Move off the RF-free ZMQ baseline onto heterogeneous hardware — CPU, GPU, and other accelerators: an O-RU over real fronthaul, GPU / accelerator offload for the PHY hot path, NIC timing for the fronthaul budget — and re-measure the knobs where the timing constraints are real.

5 · Formal reasoning — functional & non-functional

Reason about the model, not only measure it.

  • Functional — correctness and conformance: a moved boundary preserves protocol behaviour (ordering, state consistency, no deadlock).
  • Non-functional — timing and capacity: latency bounds, throughput, real-time deadline satisfaction (the PHY / FAPI budget), energy — properties proven or bounded over the actor graph.
