NextGSystem Study

Studying how to make the next-generation RAN more efficient and AI-native, on the srsRAN O-CU / O-DU stack.

srsRAN Project O-RAN OFH 7.2 CU / DU / RU split ZMQ / RF-free baseline

A study of the srsRAN Project gNB. We read the O-CU / O-DU disaggregation, find where efficiency actually lives, and test every restructuring idea as a falsifiable hypothesis against a measured baseline — rather than asserting it.

Status: bootstrap. The literature survey is largely complete (see Survey); srsRAN code analysis and baseline bring-up are next. No experiments have been run yet — hypotheses below are stated, not answered.

Operating principles

  • Evidence is falsifiable. Every efficiency claim is a hypothesis with a verifier and a numbered run record — never an assertion.
  • Point, don't duplicate. This site links to the analysis, specs, and result pages; it does not reproduce them.
  • Restructuring lives at the interfaces. F1 / E1 / FAPI / OFH are the seams the gNB is already cut along, so a proposal is "redraw this boundary."
  • Measure against a reproducible baseline. RF-free (ZMQ / RU emulator) bring-up first, so a teammate reproduces results without a radio.

1 · Architecture — the gNB, disaggregated

The srsRAN Project gNB can run monolithically or be split along O-RAN boundaries into a Central Unit (CU), a Distributed Unit (DU), and a Radio Unit (RU). The CU further splits into a control-plane (CU-CP) and a user-plane (CU-UP); the DU into a higher layer (RLC / MAC / scheduler) and a lower layer (PHY). Restructuring this project studies happens at these seams.

F1 — CU ↔ DU

The 3GPP CU/DU split (TS 38.470 family). F1-C carries control (F1AP) between CU-CP and DU; F1-U carries user data (GTP-U) between CU-UP and DU. This is the primary disaggregation boundary and the one srsRAN already supports as a split deployment.

E1 — CU-CP ↔ CU-UP

The control/user-plane separation inside the CU (TS 38.460 family, E1AP). Lets the control plane and a scaled-out user plane evolve independently — a natural place to study placement and scaling.

FAPI — MAC ↔ PHY

The Small Cell Forum interface (SCF 222) between the scheduler in DU-high and the PHY in DU-low (P5 config, P7 data). The seam where AI-RAN inference and L1 acceleration attach, under hard real-time deadlines.

Open Fronthaul 7.2x — DU ↔ RU

The O-RAN WG4 lower-layer split (7.2x) between DU-low and the RU, over eCPRI / RoE. srsRAN provides an O-RU-facing fronthaul path; the timing budget here dominates deployability.

2 · Design

Thesis. The efficiency of the RAN is governed by where the functional boundaries are drawn and how work is scheduled across them. By treating srsRAN's O-CU / O-DU split as the substrate and the F1 / E1 / FAPI / OFH interfaces as movable seams, we can study restructuring and AI-RAN integration as controlled, measurable changes rather than redesigns.

Verified working (2026-06-17). The RF-free 5G-SA testbed comes up end to end: the UE registers, gets 10.45.0.2, and pings the internet (8.8.8.8) with 0% packet loss over a pure-software radio. This is the ground every measurement in this study stands on.

# srsRAN's ZMQ RF driver links against libzmq + czmq. Build from source (into
# /usr/local) — the srsRAN-documented path, and what this baseline links against.
mkdir -p ~/src && cd ~/src
git clone https://github.com/zeromq/libzmq.git
cd libzmq && ./autogen.sh && ./configure && make -j"$(nproc)" && sudo make install && sudo ldconfig && cd ..
git clone https://github.com/zeromq/czmq.git
cd czmq  && ./autogen.sh && ./configure && make -j"$(nproc)" && sudo make install && sudo ldconfig && cd ..

2 · Build the gNB (srsRAN Project)

git clone https://github.com/srsran/srsRAN_Project.git ~/srsRAN_Project
cd ~/srsRAN_Project && git checkout release_25_10
mkdir -p build && cd build
# g++-12 + WERROR=OFF: stock gcc 9.4 dies on a -Werror warning in a bundled dep.
# ZMQ is auto-detected from the libzmq installed above — no flag needed.
cmake -DCMAKE_BUILD_TYPE=Release -DENABLE_WERROR=OFF \
      -DCMAKE_C_COMPILER=gcc-12 -DCMAKE_CXX_COMPILER=g++-12 ..
make -j"$(nproc)" gnb
# -> ~/srsRAN_Project/build/apps/gnb/gnb

7 · Run & verify

sudo ip netns add ue1                          # namespace that will hold the UE's tun

# Terminal 1 — gNB. Wait for "Cell pci=1 ... activated" and "Connected to AMF".
sudo ~/srsRAN_Project/build/apps/gnb/gnb -c ~/gnb_zmq.yaml

# Terminal 2 — srsUE. Start right after the gNB: ZMQ REQ/REP goes stale if only
# one side restarts, so always co-start the pair.
sudo ~/srsRAN_4G/build/srsue/src/srsue ~/ue_zmq.conf
# UE log: RRC Connected -> Registration Accept -> PDU Session Establishment ... IP: 10.45.0.2

# srsUE assigns the IP but not the route — add the default route, then ping out.
sudo ip netns exec ue1 ip route replace default dev tun_srsue
sudo ip netns exec ue1 ping -c4 8.8.8.8         # SUCCESS = 0% packet loss

Expected end state: tun_srsue carries 10.45.0.2/24; ping 8.8.8.8 → 4/4 received, 0% loss (RTT ~30–170 ms over ZMQ). If a step stalls, the failure is almost always one of the six in Methodology — each symptom is mapped to its fix there.

Methodology — how the bring-up was cracked

A 5G stack fails silently in a dozen places. What turned ~15 dead ends into a working link was a discipline, not luck:

  • Isolate layer by layer. Build → core health → gNB-to-AMF → RAN attach → NAS/PDU → data path. Confirm each layer's post-condition before climbing to the next.
  • Configure from the authoritative docs, not by adaptation. Pull the complete official config and reason about the protocol; a stripped-down hand-rolled config silently drops required fields.
  • Reason about the mechanism, not the symptom. "Cell selection failed" → trace MIB → CORESET#0 → SIB1. "HTTP 400 at registration" → SCP NF-discovery → PLMN mismatch.
  • Adversarially verify the fix against the docs before trusting it — this caught a wrong CORESET#0 index that would have re-created the bug.
  • Verify by observable post-condition — a RACH on the air, an assigned IP, 0% ping loss — never by "the config looks right".

1 · DSL design — model the RAN formally

The substrate every later question is asked against — two graphs and the contracts that bind them.

  • Knowledge graph — the static structure: units (CU-CP / CU-UP / DU-high / DU-low / RU), interfaces (E1 / F1 / FAPI / OFH), functions (RRC, PDCP, RLC, MAC, scheduler, PHY) and the dependencies among them. What is connected to what.
  • Actor graph — the dynamic behaviour in three primitives: events (packet arrival, TTI boundary, HARQ feedback), states (queue occupancy, HARQ buffers, channel estimate), and actions (admit a grant, encode a transport block, forward over F1). Each actor is a state machine reacting to events with actions. How the system moves.
  • Event-dispatching tree — how one event propagates to the actors that handle it: the routing hierarchy that turns a TTI boundary or a packet arrival into the ordered actions it triggers.
  • Pre/post-condition determination — the contract on each action: what must hold before it fires (pre-conditions, the guards) and what it guarantees after (post-conditions, the effects) — the hook formal reasoning (5) checks.

2 · Runtime design — the scheduling unit

The fundamental layer — the execution substrate the model runs on, and the study's crux. At what granularity is work decomposed and scheduled — a slot, a TTI, a per-UE function invocation, an actor step? The scheduling unit fixes where parallelism, pipelining, and placement across cores and accelerators can live, and is the lever restructuring actually pulls.

3 · Control knobs — the configuration space

Which states and configurations actually move the metrics: numerology / SCS, bandwidth, MCS table, the split point (monolithic / F1 / full O-RAN), scheduler policy, function placement, batching. The configuration space the study sweeps — it feeds the Performance dimensions directly.

4 · Hardware integration

Move off the RF-free ZMQ baseline onto hardware — an O-RU over real fronthaul, GPU / accelerator offload for the PHY hot path, NIC timing for the fronthaul budget — and re-measure the knobs where the timing constraints are real.

5 · Formal reasoning — functional & non-functional

Reason about the model, not only measure it.

  • Functional — correctness and conformance: a moved boundary preserves protocol behaviour (ordering, state consistency, no deadlock).
  • Non-functional — timing and capacity: latency bounds, throughput, real-time deadline satisfaction (the PHY / FAPI budget), energy — properties proven or bounded over the actor graph.
